Visión norte

La estrella que guía todo

Antes de cualquier tarea, los principios. Toda decisión técnica se mide contra estos 7 principios. Si una mejora no sirve a uno de ellos, no se hace.

"Koru es un agente secretario/amigo que debe poder hacer TODO de forma exelsa, superando siempre lo que se espera de él, tanto en contenido y calidad como en presentación UX/UI."

spa

1 · Confianza confirmada

Koru no retiene artificialmente. Crece solo con lo que el usuario confirma. Las forbiddenPhrases son ley.

fact_check

2 · Anti-alucinación

Cada dato con cita literal. Si la tool falla, se dice. Nunca se inventa. structureExtractor es referencia.

dashboard

3 · Un solo molde

KoruUnifiedCard para todo. Kicker → título → CTA. El detalle es pantalla, no card gigante. Búsqueda es estado.

bolt

4 · Interacción inmediata

Cada nueva pregunta limpia la anterior. La mascota nunca se tapa. Lo anterior vive en Historial.

memory

5 · Memoria viva

Aprende de cada interacción. Extrae con LLM, no regex. Recuerda con embeddings que siempre funcionan.

extension

6 · Tools reales

Cero placeholders. Calendario, mail, WhatsApp integrados. Si está en el catálogo, funciona.

speed

7 · Calidad exelsa

Router de modelos. JSON strict. Few-shot. Eval automática. El usuario siente que supera lo esperado.

verified

Meta-principio

Si una mejora no sirve a uno de los 6 principios anteriores, NO se hace. Mantener foco.

Dependencias críticas

Algunas tareas bloquean otras. Respetar este orden evita rework.

arrow_chain

Cadena crítica Fase 1

1.5 + 1.6 (mojibake + tildes) → 1.4 (tests) → 1.1 + 1.2 + 1.3 (limpieza UiBlock) → 1.15 (unificar routers)

Razón: los tests se arreglan después de saber qué texto busca la UI. La limpieza UiBlock se hace después de que los tests pasan. El unificar routers al final porque toca todo.

arrow_chain

Cadena crítica Fase 2

2.14 (decidir motor legacy) → 2.6 + 2.7 (particionar God-components) → 2.9 (zod) → 2.3 (extractores LLM-driven)

Razón: si se elimina el legacy, no se particiona actions.ts. zod se adopta después de particionar. Los extractores se migran al final porque necesitan zod para validar.

arrow_chain

Cadena crítica Fase 3

3.11 (backend Express) → 3.5-3.7 (calendario, mail, WA) → 3.10 (tests camino activo)

Razón: las integraciones OAuth2 necesitan un backend real (no Vite middleware). Los tests del camino activo se hacen sobre el backend extraído.

arrow_chain

Cadena crítica Fase 4

4.2 (JSON strict) → 4.3 (few-shot) → 4.1 (router modelos) → 4.4 (eval automática)

Razón: JSON strict da estabilidad. Few-shot calibra voz. Router de modelos necesita ambos estables para evaluar. Eval al final mide todo.
