    22→Stage Summary:
    23→- Koru hoy tiene **128 tools** (8 builtin + 120 externas en 11 categorías). La mitad son wrappers finos a APIs públicas sin key (Frankfurter, CoinGecko, Stooq, TheSportsDB, TheMealDB, Open Library, Wikipedia, GDELT, Open-Meteo, Nager.Date, CheapShark, lyrics.ovh, is.gd, GoQR, ZenQuotes, Free Dictionary, sunrise-sunset.org). Otra mitad son **delegates** que rebotan a `web_search`/`shopping_compare` del motor. Las más robustas (scraper + extractor anti-alucinación + chatFn Ollama) son `restaurant_deep_search` y `deep_research` (en builtins). Las más débiles son inflation_data (placeholder), ocr_text (delegate), route_plan/transport_nearby/currency_atm/weather_travel (todos delegates). Para políticas, ver `src/domain/toolRegistry.ts` (legacy 8 tools con `requiresApproval`/`autoRun`/`risk`) y `src/tools/types.ts` helpers `policies.readonly|localWrite|externalSideEffect|destructive` usados por las 120 externas.
    24→- Hay **48 variantes de UiBlock** en `types.ts` (no 15 como pregunta el usuario). `presentation/catalog.ts` sólo registra **6 visualizadores** legacy (weather, sports_scores, money_summary, news_list, comparison_table, generic_list) — son la capa intermedia que NO se usa en el flujo principal. El render real lo hace `KoruUnifiedCard` (`presentation.ts` switch de los 48 tipos → un único molde Stitch "kicker → título → metrics → CTA → detalle"). **`KoruUnifiedCard` es el "único molde"** que pide el usuario. Excepciones: `plan` (usa `PlanHeroCard`), `web_nav` y `deliverable` con `status:"working"` (no se renderizan como card — son absorbidos por el panel "Trabajando" del overlay).
    25→- `KoruDetailScreen` es la "pantalla de detalle" del flujo del usuario: se abre al clic del CTA, renderiza módulos `magical-card` con 6 kinds (text, tiles, rows, chips, scroller, timeline, sources). Sí coincide con la pantalla 4 del HTML visionario.
    26→- Memoria: 3 entidades — `MemoryFact` (candidata → confirmada → rechazada, con embedding), `Commitment` (open/done/dismissed, con recurrencia), `LifeRecord` (15 kinds: expense, medication, sleep, idea, shopping_item, deadline, recommendation, etc.). Confirmación via `confirmMemory()` en `store.ts:372` (status → "confirmed", confidence ≥ 0.9, +18 trustedEnergy). Búsqueda semántica en `memory/semanticSearch.ts` usa cosine sim (umbral 0.55) sobre embeddings guardados; embeddings se generan con Ollama local (`shared/embeddings.ts`). Extractores automáticos en `knowledge/extract*` corren por cada turno: `extractMemoryCandidatesFromText`, `extractCommitmentsFromText`, `extractLifeRecordsFromText` (regex/determinísticos, sin LLM).
    27→- Flujo de un turno (7 pasos): (1) `TalkOverlay` envía texto → (2) `KoruProvider.submitEntry` → (3) `runBackendAgentTurn` POST a `/api/koru/turn?stream=true` → (4) `runKoruBackendTurn` corre Semantic Router → si es `deep_research` o "deliverable topic" explícito → `runDeepResearchFlow` (pipeline con 6 fases + chunks de progreso real); si no → callProvider con tools, ejecuta tools, segundo call para síntesis JSON → (5) NDJSON stream de chunks (`reply`, `uiBlocks`, `stateEvents`) → (6) `KoruProvider` merge items por tipo (preserva ids), setPhase → (7) `applyBackendTurnToState` persiste memorias/records/commitments. La decisión "Trabajando..." vs inmediato está en `KoruProvider.runReadonlyWebAction` (1187-1203: `depth: "deep"|"quick"`) y en `TalkOverlay.workingDeliverable` (si último turno de Koru tiene un `deliverable` con `status:"working"`). El formato de card se decide en `chatCards.tsx:585-615` (3 casos especiales + default → `KoruUnifiedCard`). El stream es NDJSON con timeout de inactividad (no total) en `backendAgentClient.ts:64-68`.
    28→- Persistencia: **100% client-side**, no hay DB backend. `persistence.ts` usa IndexedDB (`koru-local-first`, store `state`, key `current`) como primario + localStorage key `koru.mvp.state.v1` como legacy/mirror. `store.ts:saveState` escribe ambos. Todo el `KoruState` (memorias, commitments, records, entries, energyEvents, nudges, modelCalls) sobrevive entre sesiones. El server es stateless — sólo mantiene providers en memoria.
    29→- Comparación con el HTML visionario (5 pantallas): **4 de 5 ya existen**. Pantalla 1 (pedido + susurro) ✓ existe — el "susurro" es el `setActivity(inferActivity)` en `KoruProvider`. Pantalla 2 (Trabajando) ✓ existe — `WorkingPanel` con progreso real del pipeline deep_research. Pantalla 3 (carta entregada) ✓ existe — es el bloque `deliverable` con `status:"ready"` renderizado por `KoruUnifiedCard`. Pantalla 4 (detalle) ✓ existe — `KoruDetailScreen` con módulos `magical-card`. Pantalla 5 (nueva pregunta limpia anterior) ✓ existe — `TalkOverlay.visibleTurns` sólo muestra el último intercambio usuario→Koru. Lo que aún NO está al 100%: las "45 cards" del prompt del HTML original son en realidad 48 variantes, y no todas se normalizan limpiamente — algunas (live_match, market, crypto_portfolio, forex, election_results, route_map, birthday_calendar) tienen datos muy específicos que el `toPresentation` del `presentation.ts` aplana al molde hero/métricas/CTA pero pierden riqueza visual específica. Las 4 reglas del footer del HTML visionario ("un solo molde", "detalle es pantalla", "búsqueda es estado no card", "interacción inmediata no chat") están **implementadas en código** pero el catálogo de visualizadores legacy (6 entradas en `catalog.ts`) sigue presente y podría ser fuente de confusión.

   421→### C.2 Confirmación de memorias (candidate → confirmed)
   422→
   423→Implementado en `src/domain/store.ts:372-398`:
   424→
   425→```ts
   426→export function confirmMemory(state: KoruState, id: string): KoruState {
   427→  const now = nowIso();
   428→  const next = {
   429→    ...state,
   430→    memories: state.memories.map((memory) =>
   431→      memory.id === id ? {
   432→        ...memory,
   433→        status: "confirmed" as const,
   434→        confidence: Math.max(memory.confidence, 0.9),
   435→        confirmedAt: now,
   436→        updatedAt: now,
   437→        useForSuggestions: memory.useForSuggestions ?? memory.sensitivity === "normal",
   438→      } : memory,
   439→    ),
   440→    trustedEnergy: state.trustedEnergy + 18,
   441→    updatedAt: now,
   442→  };
   443→  next.stage = stageFor(next);
   444→  saveState(next);
   445→  return next;
   446→}
   447→```

Cada confirmación suma **+18 trustedEnergy** (sistema de gamificación que controla `KoruStage`: seed → sprout → roots → born → garden).

   451→### C.3 Búsqueda semántica
   452→
   453→Implementado en `src/domain/memory/semanticSearch.ts` (28 líneas):
   454→
   455→```ts
   456→export async function selectRelevantMemoriesSemantic(
   457→  input: string,
   458→  memories: MemoryFact[],
   459→  embedFn: (text: string) => Promise<number[]>,
   460→  maxResults = 5,
   461→): Promise<MemoryFact[]> {
   462→  const queryVector = await embedFn(input);
   463→  return memories
   464→    .filter((m) => m.status !== "rejected" && m.embedding?.length)
   465→    .map((m) => ({ memory: m, score: cosineSimilarity(queryVector, m.embedding ?? []) }))
   466→    .filter((item) => item.score >= 0.55)  // umbral
   467→    .sort((a, b) => b.score - a.score)
   468→    .slice(0, maxResults)
   469→    .map((item) => item.memory);
   470→}
   471→```
   472→
   473→- **Embeddings locales vía Ollama** — ver `src/tools/shared/embeddings.ts` y `src/domain/memory/embeddings.ts`. La función `makeEmbedFn()` produce un callback que llama a `http://localhost:11434/api/embeddings` con modelo `nomic-embed-text` (o el configurado).
   474→- `isOllamaAvailable()` en `shared/embeddings.ts` hace un ping a Ollama antes de usarlo.
   475→- La `memory_search` tool (en `tools/knowledge/knowledge.ts:49-120`) usa `selectRelevantMemoriesSemantic` con fallback léxico (token overlap) si Ollama no está disponible.
   476→- Umbral de similitud: **0.55** (cosine). Si no hay match, cae a búsqueda léxica por tokens normalizados (sin acentos, split por espacios, tokens > 2 chars).
   477→- Los embeddings se guardan en `MemoryFact.embedding` al crear la memoria (en `store.ts:712-718` — función `ensureMemoryEmbedding`).

   508→## D. Flujo de un turno
   509→
   510→### D.1 Los 7 pasos desde `/api/koru/turn` hasta el render
   511→
   512→**Paso 1 — UI captura input** (`TalkOverlay.tsx`):
   513→- Usuario escribe en input → `handleTextSubmit` → `submitText` → `sendMessage(clean, "typed")`.
   514→- `sendMessage` agrega el user turn a `chatTurns`, llama a `submitEntry`.
   515→
   516→**Paso 2 — Frontend prepara payload** (`KoruProvider.tsx:905-927`):
   517→- `submitEntry(text, source, history)` setea `processing=true`, construye `domainHistory` (últimos 12 turnos con texto), llama a `runBackendAgentTurn(text, previousState, history, model, onChunk)`.
   518→
   519→**Paso 3 — POST con stream NDJSON** (`backendAgentClient.ts:54-120`):
   520→- POST a `/api/koru/turn` con `{ input, state, history, stream: true, model }`.
   521→- Timeout de **inactividad** (no total): cada chunk resetea el timer (líneas 64-68). Permite que un deep_research de 3-4 min siga vivo mientras streamee progreso.
   522→- Response `Content-Type: application/x-ndjson` → lee línea por línea, parsea JSON, llama `onChunk(chunk)`.
   523→
   524→**Paso 4 — Backend decide ruta** (`koruBackend.ts:3881-4015`):
   525→- `runKoruBackendTurn(request, config, onChunk)`.
   526→- **Primera bifurcación**: `explicitDeliverableTopic(inputTrimmed)` → si el input pide explícitamente un informe/análisis → `runDeepResearchFlow` (pipeline dedicado, 6 fases con progreso real).
   527→- **Segunda bifurcación**: Semantic Router (`getRouter(config).route(input)`) clasifica en categorías (weather, world_info, shopping, planning, personal_query, action, research, sports, market, travel, directions, elections, review, birthday, conversation). Si el router detecta tool con confianza ≥ 0.78 (umbral extra para `web_search`) → ejecuta tool directo sin llamar al LLM primero.
   528→- Si router dice `deep_research` → también va a `runDeepResearchFlow`.
   529→- Si no hay router o dice `conversation` → flujo nativo.
   530→
   531→**Paso 5a — Flujo nativo LLM** (`koruBackend.ts:4024-4120`):
   532→- `callProvider(config, messages, timeout, toolsEnabled=true, preferredProvider, filteredTools)` con tools filtradas por `CATEGORY_TOOLS[routeCategory]`.
   533→- Si el LLM pide tool_calls → emite chunk "Buscando..." con `web_nav loading` → `executeProviderToolCalls` (dispatch a `TOOL_BOX` o builtins) → chunk intermedio con resultados → segundo `callProvider` SIN tools para síntesis JSON.
   534→- El JSON final debe tener: `reply, understanding, uiBlocks, suggestedActions, memoryCandidates, commitments, records, sentiment`.
   535→- Las `deferredDataCards` (promesas de `data_card` validado) se resuelven **en paralelo** con el segundo call (líneas 4102-4106) para no sumar latencia.
   536→
   537→**Paso 5b — Flujo deep_research** (`koruBackend.ts:3676-3879`):
   538→- `emit(6, "Entendiendo el pedido…", "thinking", ...)` → chunk con `deliverable status:working progress:6`.
   539→- Planificación de sub-búsquedas con LLM (pide JSON `{"queries":[...]}`).
   540→- Loop de búsquedas web reales: `emit(12 + i/total*43, "Buscando fuentes X/Y…", "searching")`.
   541→- `emit(62, "Comparando y cruzando fuentes…", "comparing")`.
   542→- `emit(78, "Redactando tu informe…", "planning")` + síntesis LLM con prompt estructurado (title, description, summary, categories[3], metrics[3], sections[4-6]).
   543→- Devuelve `deliverable status:ready progress:100` con sections, sources, memoryCandidate "Le interesa ${topic}".
   544→
   545→**Paso 6 — Streaming al frontend** (`KoruProvider.tsx:927-997`):
   546→- `onChunk(chunk)` se llama por cada NDJSON line.
   547→- **Primer chunk** → crea `koruTurn` nuevo con `status: "working"` y `mascotState: chunk.mascotState ?? "working"`. Items = `blocksToItems(chunk.uiBlocks)` (cada block → item con `kind: "action"`, `status: "working"`).
   548→- **Chunks siguientes** → merge por TIPO de block (no por índice, para preservar identidad de React y no reiniciar animaciones). Si llega un block del mismo tipo que uno existente, lo reemplaza conservando el id.
   549→- `chunk.stateEvents` → `setPhase(lastEvent.kind)` para el WorkingPanel.
   550→- **Chunk final** (isDone = `stateEvents.some(e => e.kind === "done")`) → `status: "done"`, `mascotState: chunk.mascotState ?? "idle"`.
   551→
   552→**Paso 7 — Persistencia y render final** (`KoruProvider.tsx:998-1048`):

   593→### D.4 Streaming
   594→
   595→**Protocolo**: NDJSON (Newline-Delimited JSON) sobre HTTP/1.1 chunked.
   596→
   597→**Server** (`vite.config.ts:870-884`):
   598→```ts
   599→if (streamEnabled) {
   600→  res.statusCode = 200;
   601→  res.setHeader("Content-Type", "application/x-ndjson");
   602→  res.setHeader("Cache-Control", "no-cache");
   603→  res.setHeader("Connection", "keep-alive");
   604→  const onChunk = (chunk) => {
   605→    const line = JSON.stringify(chunk) + "\n";
   606→    chunks.push(line);
   607→    res.write(line);
   608→  };
   609→  const result = await runKoruBackendTurn(request, config, onChunk);
   610→  const line = JSON.stringify(result) + "\n";
   611→  if (!chunks.includes(line)) res.write(line);
   612→  res.end();
   613→}
   614→```
   615→
   616→**Client** (`backendAgentClient.ts:79-107`):
   617→- `response.body.getReader()` → `TextDecoder` → buffer acumulado → split por `\n` → `JSON.parse(line)` → `onChunk(chunk)`.
   618→- Cada chunk es un `KoruBackendTurnResponse` completo (con `reply`, `uiBlocks`, `stateEvents`, etc.) — el cliente hace merge incremental.

   622→## E. Persistencia
   623→
   624→### E.1 Dónde se guarda todo
   625→
   626→**`src/domain/persistence.ts`** (103 líneas) — doble capa:
   627→
   628→1. **IndexedDB** (primario):
   629→   - DB name: `koru-local-first`, versión 1.
   630→   - Object store: `state` (keyPath: `key`).
   631→   - Key única: `current` (snapshot completo del KoruState).
   632→   - Operaciones: `readPersistedState()`, `writePersistedState(state)`, `clearPersistedState()`.
   633→
   634→2. **localStorage** (legacy/mirror, `koru.mvp.state.v1`):
   635→   - `readLegacyState()`, `writeLegacyState(state)`, `clearLegacyState()`.
   636→   - Se mantiene como cache de fallback si IndexedDB falla.
   637→
   638→`saveState` (`store.ts:167-171`) escribe en **ambos**:
   639→```ts
   640→export function saveState(state: KoruState): void {
   641→  const snapshot = normalizeState({ ...state, updatedAt: nowIso() });
   642→  writeLegacyState(snapshot);
   643→  void writePersistedState(snapshot).catch(() => undefined);
   644→}
   645→```
   646→
   647→### E.2 ¿Hay backend de base de datos?
   648→
   649→**No.** El server (Vite middleware en `vite.config.ts`) es **stateless**:
   650→- No tiene DB.
   651→- No persiste nada entre requests.
   652→- Sólo mantiene en memoria: `config` (API keys de proveedores) y el router (que se cachea en module-level).
   653→- Todo el estado del usuario viaja en cada request (`request.state`) y se devuelve actualizado en cada response.
   654→
   655→**Excepción**: logs en `logs/koru-turns.jsonl` (server-side, solo para debugging/auditoría) y `/koru-audit/log` (cliente-side, posting events de auditoría del cliente al server).
   656→
   657→### E.3 ¿Las memorias/commitments/records sobreviven entre sesiones?
   658→
   659→**Sí**, totalmente. Como todo el `KoruState` se guarda en IndexedDB + localStorage:
   660→- Cerrar el browser y reabrir → el estado se recupera con `readPersistedState()` al inicializar el provider.
   661→- Refresh de la página → mismo mecanismo.
   662→- Cambiar de dispositivo/p navegador → NO se sincroniza (es local-first puro, sin cloud).
   663→
   664→**Excepción**: modo ephemeral (`ephemeralMode: true`) — guarda el snapshot pero SIN memorias/commitments/records del turno actual (ver `saveEphemeralSessionSnapshot` en `store.ts:173-189` que preserva el estado previo).
